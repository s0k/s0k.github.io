Git 笔记

Posted at — Jun 15, 2019

Git基础设置

设置签名（作用是区分每个人） 1.设置用户名 Git config --global user.name 'rab3it'

2.设置用户名邮箱 Git config --global user.email 'rab3it@qq.com'

设置的签名和github账号没关系（可以不一样）

签名级别 1.项目级别/仓库级别：仅在本地库范围内有效 git config user.name/email tom

2.系统用户级别：登录当前操作系统的用户范围 git config --global user.name/email tom

注： .git文件为隐藏文件，里面有config文件，储存着用户名和邮箱以及密码 比如：url = git@github.com:rab3it/rab3it.github.io.git

初始化一个新的Git仓库

1.创建文件夹 mkdir test #新建文件夹

2.在文件夹内初始化Git（创建Git仓库） cd test git init #初始化（初始化后会在本地创建一个隐藏的.git文件）

向仓库提交文件

1.创建a.php文件 touch a.php

Git 工作区域
Git仓库 (Git Repository)
暂存区
工作区（Working Dictory）    #（包含.git文件的目录或者说包含本地仓库的目录）

文件流程：工作区 -- 暂存区 -- Git仓库

2.查看git状态（查看当前文件在哪个区域以及文件状态） git status

3.提交工作区文件到暂存区 git add a.php #git add 文件 git add -A #提交所有文件 注： 从暂存区撤回：git rm --cached a.php #git rm –cached 文件

4.提交暂存区文件到Git区 git commit -m "add a.php" #git commit -m “提交描述” （m message）

分支

在版本控制过程中，使用多条线同时推进多个任务

master（主干）

新开辟分支： feature_blue（前端模块） feature_game（游戏模块） 最后合并分支

分支的好处： 同时并行推进多个功能开发，提高开发效率 各个分支在开发过程中，如果某一个分支开发失败，不会对其他分支有任何影响。失败的分支删除重新开始即可。

分支操作

git status 结果有master，默认主干为master

查看分支 git branch -v

添加分支 git branch hot_fix #git branch 分支名

切换分支 git checkout hot_fix #git checkout 分支名

git branch -v #发现已经在hot_fix分支

hot_fix修改好后进行合并分支

合并分支

1.切换到接受修改的分支（被合并，增加新内容）上， 2.执行merge命令

git branch -v
git checkout master
git branch -v
git merge hot_fix

合并分支产生冲突

master和hot_fix 同时修改一个文件

提示手动修改 git status

解决冲突： 1.编辑文件，删除特殊符号 2.把文件修改到满意的程度 3.git add 文件名 4.git commit -m ‘日志信息’ #此时commit不要带文件名

Git管理远程仓库

一个服务器，多人共同开发

本地仓库同步到Git远程仓库中

Git 克隆操作

目的：将远程仓库（GitHub对应的项目）复制到本地

git config --list #列出所有 Git 当时能找到的配置

代码： git clone https://github.com/rab3it/rab3it.github.io.git #git clone 仓库地址

给本地远程库起别名

查看别名 get remote -v

添加别名 git remote add origin https://github.com/rab3it/rab3it.github.io.git #origin 是远程库（后面的链接）的别名， add 添加

Git推送

推送 origin（远程库的别名） git push origin master

远程库修改的拉取

pull = fetch + merge`

抓取远程库文件 git fetch origin master #git fetch 远程库地址 远程分支名

#远程合并到本地 git merge origin/master #git merge 远程库地址/远程分支名

直接执行pull(条件 不会产生冲突的情况下) git pull origin master

解决团队协作会冲突 要点： 1.如果不是基于GitHub远程库的最新版所做的修改，不能推送，必须先拉取。 2.拉取下来后如果进入冲突状态，则按照“分支冲突解决”操作解决即可。
